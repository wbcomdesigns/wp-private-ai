# WP Ability Scanner

Scans any WordPress plugin and generates a ready-to-drop-in [WP Abilities API](https://developer.wordpress.org/abilities-api/) adapter — without touching the original plugin.

Requires PHP 8.1+. No Composer. No dependencies.

---

## Usage

```bash
# Print adapter to stdout
php wp-ability-scanner.php /path/to/plugin

# Write adapter to a file
php wp-ability-scanner.php /path/to/plugin --output=woocommerce-abilities.php
```

Drop the generated file into `wp-content/mu-plugins/` on any WordPress 6.9+ site.

---

## What It Detects

| Source | Example |
|---|---|
| REST routes (string literal) | `register_rest_route('wc/v3', '/products', [...])` |
| REST routes (WP_REST_Server constants) | `'methods' => WP_REST_Server::READABLE` |
| Custom post types | `register_post_type('product', [...])` |
| Public CRUD methods | `public function get_contacts(...)` |

### WP_REST_Server Constant Mapping

| Constant | HTTP Methods |
|---|---|
| `READABLE` | `GET` |
| `CREATABLE` | `POST` |
| `EDITABLE` | `PUT, PATCH` |
| `DELETABLE` | `DELETE` |
| `ALLMETHODS` | `GET, POST, PUT, PATCH, DELETE` |

## Implementing the TODO Callbacks

Every generated ability has a `TODO` in its `execute_callback`. Implement it using the plugin's own PHP API:

### REST route ability

The comment shows the endpoint: `// Endpoint: GET /wp-json/wc/v3/products`

```php
'execute_callback' => function( array $input ): array|\WP_Error {
    $response = WC()->api->get_endpoint_data( '/wc/v3/products', $input );
    if ( is_wp_error( $response ) ) {
        return $response;
    }
    return $response;
},
```

Or use an internal WooCommerce data store directly:

```php
'execute_callback' => function( array $input ): array|\WP_Error {
    $args = [
        'status'   => 'publish',
        'limit'    => $input['per_page'] ?? 10,
        'page'     => $input['page'] ?? 1,
        'search'   => $input['search'] ?? '',
    ];
    $products = wc_get_products( $args );
    return array_map( fn( $p ) => [
        'id'    => $p->get_id(),
        'name'  => $p->get_name(),
        'price' => $p->get_price(),
    ], $products );
},
```

### CPT ability

The comment shows the post type: `// Endpoint: WP_Query CPT: product`

```php
'execute_callback' => function( array $input ): array|\WP_Error {
    $query = new WP_Query( [
        'post_type'      => 'product',
        'posts_per_page' => $input['per_page'] ?? 10,
        'paged'          => $input['page'] ?? 1,
        's'              => $input['search'] ?? '',
    ] );
    return array_map( fn( $p ) => [
        'id'    => $p->ID,
        'title' => $p->post_title,
        'slug'  => $p->post_name,
    ], $query->posts );
},
```

### CRUD method ability

The comment shows the class and method: `// Endpoint: WC_Product_Data_Store_CPT::get_products()`

```php
'execute_callback' => function( array $input ): array|\WP_Error {
    $store = WC_Data_Store::load( 'product' );
    return $store->get_products( $input );
},
```

## Ability Naming Rules

Generated ability names follow WordPress core requirements:

- Format: `plugin-slug/action-resource`
- Only: lowercase letters, numbers, dashes, slashes
- Examples: `woocommerce/get-products`, `fluent-crm/list-contacts`

HTTP method → action mapping:

| HTTP | Action |
|---|---|
| `GET` | `get` |
| `POST` | `create` |
| `PUT` / `PATCH` | `update` |
| `DELETE` | `delete` |

---

## API Hooks (WP 6.9+)

| Hook | Purpose |
|---|---|
| `wp_abilities_api_categories_init` | Register categories |
| `wp_abilities_api_init` | Register individual abilities |

The scanner uses both correct hooks. Do **not** use `init` — the Abilities API has its own dedicated hooks.
